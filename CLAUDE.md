## Commands

```sh
bun install                    # install backend deps
cd web && bun install          # install frontend deps (separate package.json)

# Run the CLI (single turn)
bun run src/index.ts "<prompt>"
bun run src/index.ts --session <id> --hide-reasoning "<prompt>"
bun run src/index.ts --project <name> "<prompt>"   # auto-creates the project if missing

# Run the web UI (two processes)
bun run serve                  # Bun HTTP + SSE on :3000
cd web && bun run dev          # Vite dev server on :5173, proxies /api

# Tests
bun test                              # full suite
bun test tests/agent/render.test.ts   # single file
bun test -t "closes reasoning block"  # single test by name
bun test --watch

bun run typecheck              # tsc --noEmit
bun run fmt                    # prettier
bun run check                  # typecheck + test
bun run docs                   # TypeDoc ŌåÆ docs/api/

bun run web:build              # build web/dist/ (Bun then serves it on :3000 in prod)
bun run build                  # compile standalone binary via scripts/build.ts (all platforms + Tauri)
bun run build:platform darwin-arm64  # single-platform build
bun run build -- --no-web      # skip Vite build, reuse existing web/dist/
bun run build -- --no-client   # skip Tauri client build (useful without Rust toolchain)
bun run build -- --list        # list available build targets

# Tauri desktop client
cd client && bun install       # install client deps (separate package.json, requires Rust toolchain)
bun run client:dev             # Tauri dev mode (opens native window)
bun run client:build           # Tauri production build (platform-specific installer)
```

**Runtime:** Bun Ōēź 1.3.0. Node is not supported ŌĆö the project relies on `bun:sqlite`, `Bun.serve`, `Bun.TOML`, `bun:test`.

**State:** everything lands in `./.bunny/` relative to cwd (override via `$BUNNY_HOME`). Portable by design ŌĆö no `$HOME/.config`. LLM/embed API keys come from env (`LLM_API_KEY`, `EMBED_API_KEY`); user-facing API keys (`BUNNY_API_KEY` for the CLI) are minted per user via the web UI. Seeded-admin credentials can be overridden with `BUNNY_DEFAULT_ADMIN_PASSWORD`. Project-level choices go in `bunny.config.toml` (incl. `[auth]`, `[web]`, `[translation]`, `[telegram]`, `[code]`, `[prompts]` blocks).

## Architecture

The agent loop is a thin outer/inner loop (Mihail Eric, _The Emperor Has No Clothes_). Three design principles:

1. **Minimal agent loop** ŌĆö `src/agent/loop.ts:runAgent` is the only orchestrator: build system prompt (with hybrid recall injected) ŌåÆ stream LLM ŌåÆ if tool_calls, execute in parallel ŌåÆ repeat until assistant answers without tools. Capped at `MAX_TOOL_ITERATIONS = 20`.
2. **Queue is the spine** ŌĆö every meaningful action is a fire-and-forget job on a `bunqueue` worker (`src/queue/`) which logs to `events` in SQLite. Covers LLM req/res, tool calls/results, memory writes, **and** every HTTP mutation. `LogPayload` carries an optional `userId` so every event is attributable. Every route context (`AuthRouteCtx`, `WorkspaceRouteCtx`, `AgentRouteCtx`, `ScheduledTaskRouteCtx`, `BoardRouteCtx`) includes `queue: BunnyQueue`. Nothing is invisible; nothing blocks the caller. See [ADR 0004](./docs/adr/0004-bunqueue-as-spine.md), [docs/dev/concepts/queue-and-logging.md](./docs/dev/concepts/queue-and-logging.md).
3. **Portable state** ŌĆö single SQLite file under `$BUNNY_HOME`. Schema in `src/memory/schema.sql` ŌĆö **append-only; NEVER drop or rename columns**. Key tables: `messages`, `projects`, `agents`, `project_agents`, `skills`, `project_skills`, `board_swimlanes`, `board_cards`, `board_card_runs`, `scheduled_tasks`, `whiteboards`, `documents`, `contacts`, `contact_groups`, `contact_group_members`, `businesses`, `contact_businesses`, `business_translations`, `kb_definitions`, `code_projects`, `users`, `auth_sessions`, `api_keys`, `session_visibility`, `events`, `notifications`, `messages_fts` (FTS5). The `embeddings` vec0 table is created dynamically because the dimension must be baked into the CREATE statement. See [ADR 0003](./docs/adr/0003-sqlite-fts5-vec-hybrid.md), [docs/dev/reference/data-model.md](./docs/dev/reference/data-model.md).

### Streaming and renderer

`src/llm/adapter.ts:chat` returns `{ deltas, response }`; SSE parser in `src/llm/stream.ts` is multi-byte safe; **provider-specific streaming quirks belong in `src/llm/profiles.ts`**, not in `adapter.ts` or `stream.ts`. Reasoning is stored on `messages.channel='reasoning'` but not replayed to the LLM, except anthropic-compat providers that need the thinking-block signature roundtrip (`provider_sig` column). `src/agent/render.ts` defines the `Renderer` interface (`onDelta`, `onToolResult`, `onError`, `onTurnEnd`, optional `onAskUserQuestion?` / `onQueueWait?` / `onQueueRelease?`) ŌĆö implemented by `createRenderer` (ANSI) and `createSseRenderer` (`src/agent/render_sse.ts`). **When adding a front-end, implement this interface ŌĆö do not branch inside `loop.ts`.** **Upstream concurrency gate:** `chat()` is throttled by a process-wide semaphore in `src/llm/concurrency_gate.ts` (`getGlobalGate(cap)`); cap comes from `cfg.llm.maxConcurrentRequests` (`[llm] max_concurrent_requests`, env `LLM_MAX_CONCURRENT_REQUESTS`, default **1**). When a call has to wait the renderer's `onQueueWait?` fires (frontend shows "In wachtrij (positie X)"), and `onQueueRelease?` fires when the gate releases ŌĆö the SSE pair `llm_queue_wait` / `llm_queue_release` drives the chat-tab badge and pauses the elapsed-time timer (reuses the `pausedAtMs` / `pausedTotalMs` mechanism). See [ADR 0002](./docs/adr/0002-openai-compat-adapter.md), [ADR 0005](./docs/adr/0005-streaming-reasoning.md), [ADR 0035](./docs/adr/0035-llm-concurrency-gate.md), [docs/dev/concepts/streaming-and-renderers.md](./docs/dev/concepts/streaming-and-renderers.md).

### Memory

`src/memory/recall.ts:hybridRecall` fuses BM25 (FTS5 trigram) + kNN over embeddings (`sqlite-vec`) with Reciprocal Rank Fusion (k=60). Top-`recall_k` (default 8) messages injected into the system prompt. Embedding failures are non-fatal ŌĆö the code still runs without `EMBED_API_KEY`. Short-term coherence: the last `memory.last_n` (default 10) user/assistant *content* turns of the current session are replayed verbatim (`getRecentTurns` in `src/memory/messages.ts`, spliced between system prompt and new user message). Tool-call/tool-result/reasoning rows are skipped. Replayed ids flow to `hybridRecall` via `excludeIds` so recall never duplicates. `last_n = 0` ŌåÆ recall-only. `last_n` / `recall_k` overridable per project via `systemprompt.toml` (`lastN` / `recallK` in the DTO). `messages` rows are one-per-semantic-unit (`channel` column); FTS triggers fire on `channel='content'` only. See [docs/dev/concepts/memory-and-recall.md](./docs/dev/concepts/memory-and-recall.md).

### Memory & Soul (per-user / per-agent)

Persistent text bodies that ride alongside hybrid recall. **Hybrid recall is _episodic_; memory + soul are _persistent_.** Three rows per scope, each capped at 4 000 chars, auto-curated hourly. Tables: `user_project_memory(user_id, project)`, `agent_project_memory(agent, project)`, plus seven `soul_*` columns on `users` (`users.soul` is global per-user). Each carries a `watermark_message_id` so the analyser only sees messages past the last successful refresh ŌĆö no fixed window. `status` machine `idle ŌåÆ refreshing ŌåÆ (idle | error)` mirrors `kb_definitions`; stuck rows reclaimed by the same handler at the start of every tick (`refreshing_at < now - cfg.memory.refreshStuckThresholdMs`). Scheduled handler `memory.refresh` cron `0 * * * *` walks active pairs (CTE ordered by gap-size DESC), capped at `cfg.memory.refreshBatchSize` LLM calls per tick across all three passes. Read-side: `runAgent` calls `loadMemoryContext(db, { userId, project, agent })` once per turn ŌåÆ spliced via `BuildSystemMessageOpts.{userSoul,userMemory,agentProjectMemory}` into a `## Persistent context` block. **Bypasses every `systemPromptOverride` path** so KB generate, doc/code edit, and the refresh handler itself stay deterministic and the system-user's soul never feeds back into a curation run. Manual edits are seeds, not locks: the prompt instructs the LLM to keep every existing fact unless contradicted, and to rewrite-to-budget when the merged body would exceed 4 000 chars (write-time `clampMemory` is the belt). HTTP: `GET/PUT /api/users/me/soul`, `GET/PUT /api/projects/:project/memory/me`, `GET/PUT /api/projects/:project/memory/agents/:agent`, admin-only `GET .../users/:userId`. Three new prompt-registry entries (`memory.user_project.refresh`, `memory.agent_project.refresh`, `memory.user_soul.refresh`). Queue topic: `memory`. Entry: `src/memory/{user_project_memory,agent_project_memory,refresh_handler,refresh_helpers}.ts`, `src/server/memory_routes.ts`, `web/src/tabs/MemoryPanel.tsx`. See [ADR 0034](./docs/adr/0034-per-user-agent-memory.md), [docs/dev/concepts/memory-and-soul.md](./docs/dev/concepts/memory-and-soul.md).

### Web UI

`bunny serve` (`src/server/index.ts`) routes `/api/*` through `src/server/routes.ts` (plain pathname switch, no framework); serves `web/dist/` if present, falls back to embedded `webBundle`, then dev placeholder; `idleTimeout: 0` so SSE streams can outlive the default. Frontend: React + Vite (separate `web/package.json`). Permanent **56 px left icon-rail** (`web/src/components/Sidebar.tsx`) expanding to 240 px on hover (VS Code pattern ŌĆö no layout reflow). Nav groups: **Work** (Chat, Board, Tasks), **Content** (Documents, Whiteboard, Files, Code, Contacts, Knowledge Base, News), **Configure** (Workspace), **System** (Dashboard, Settings). Below 640 px ŌåÆ hamburger. **Icons go through the barrel at `web/src/lib/icons.ts` ŌĆö never import `lucide-react` directly.** Rabbit mascot = brand logo + 0.04-opacity `.app-shell__main` watermark (skipped on Dashboard via `.app-shell__main--dense`) + `<EmptyState>` illustrations + login hero. **Visual language is canonised in [`docs/styleguide.md`](./docs/styleguide.md) ŌĆö consult before adding UI.**

SSE event shapes live in `src/agent/sse_events.ts`, imported by both `src/agent/render_sse.ts` (backend) and `web/src/api.ts` (frontend) ŌĆö single source of truth, compile-time drift guard (Vite's `server.fs.allow: [".."]` permits the cross-root import). Chat uses `fetch` body-reader, **not** `EventSource` (POST JSON). App boots via `GET /api/auth/me` ŌĆö 401 ŌåÆ login, `mustChangePassword` flag ŌåÆ forced-change page. All fetches use `credentials: "include"`.

`localStorage` keys: `bunny.activeTab` (legacy `messages`/`logs`/`projects`/`agents`/`skills` aliased forward via `LEGACY_TAB_ALIAS` in `App.tsx`), `bunny.activeSessionId`, `bunny.activeProject`, `bunny.activeAgent.<sessionId>`, `bunny.webNews.template`, `bunny.activeCodeProject.<project>`, `bunny.activeCodeFeature`, `bunny.codeChatSession.<codeId>`. **Switching project always starts a new session.** See [ADR 0006](./docs/adr/0006-web-ui.md), [ADR 0020](./docs/adr/0020-ui-redesign-and-styleguide.md), [ADR 0031](./docs/adr/0031-every-chat-is-agent-bound.md), [docs/dev/ui/](./docs/dev/ui/).

## Tool-gating flags (read before touching a handler)

Some tools are closure-bound per run (`DYNAMIC_TOOL_NAMES` in `loop.ts`) and spliced into the registry by `buildRunRegistry`. Three flags on `RunAgentOptions` gate user-facing side effects. Getting these wrong produces silent hangs or duplicate notifications.

| Flag              | Enabled only by                                                 | Reason                                                                                          |
| ----------------- | --------------------------------------------------------------- | ----------------------------------------------------------------------------------------------- |
| `askUserEnabled`  | `POST /api/chat`, `POST /api/messages/:id/regenerate`           | Background renderers (edit modes / card runs / scheduler ticks) can't surface the question card |
| `mentionsEnabled` | `POST /api/chat` only                                           | Re-runs and background work must not re-notify                                                  |
| `telegramCfg`     | `POST /api/chat` only                                           | Same ŌĆö mirrored from `mentionsEnabled` gating                                                   |

`webCfg` is a config, not a gate: pass it wherever web tools should splice in (chat, regenerate, KB definition generate, Web News run, Code edit/chat).

## Column invariants (append-only, never rename)

- `messages.author` ŌĆö responding agent name. New turns are never NULL (the `/api/chat` route falls back to `cfg.agent.defaultAgent`, seeded + auto-linked at boot by `src/memory/agents_seed.ts`). Legacy NULL rows remain in history and re-label at render time via `resolveBubbleLabel`. Append-only ŌĆö see [ADR 0031](./docs/adr/0031-every-chat-is-agent-bound.md).
- `messages.project` ŌĆö legacy NULL rows read back as `'general'` via `COALESCE(project, 'general')`. A session is locked to one project ŌĆö `runAgent` errors on mismatch.
- `messages.regen_of_message_id` ŌĆö chains regenerate-as-alt-version.
- `messages.provider_sig` ŌĆö anthropic-compat thinking-block signature roundtrip.
- `<entity>.deleted_at` / `deleted_by` ŌĆö soft-delete marker for `documents`, `whiteboards`, `contacts`, `kb_definitions`, `code_projects`. Non-null = in Trash. Board cards use `archived_at` (separate flow).
- `<entity>.original_lang` / `source_version` ŌĆö translation state; every source-field edit path calls `markAllStale`.

## Projects

Logical workspaces that group sessions/messages + on-disk assets. Metadata lives in the `projects` table (name PK, description, visibility, created_by, languages, default_language, timestamps); system prompt lives on disk at `$BUNNY_HOME/projects/<name>/systemprompt.toml` (`prompt`, `append`).

**Default project name + base system prompt** come from `[agent]` in `bunny.config.toml` (`default_project`, `system_prompt`, `default_agent`), env overrides `BUNNY_DEFAULT_PROJECT` / `BUNNY_SYSTEM_PROMPT` / `BUNNY_DEFAULT_AGENT`. `runAgent` takes them via `RunAgentOptions.agentCfg`; `buildSystemMessage` falls back to a hard-coded prompt only in tests. CLI boot (`src/index.ts`) and `startServer` seed the configured default on top of the always-present `general` row, then `ensureDefaultAgent` seeds the configured default agent and links it to every project.

Entry points: `src/memory/projects.ts` (CRUD + `validateProjectName` + `getSessionProject`), `src/memory/project_assets.ts` (`ensureProjectDir`, `loadProjectSystemPrompt`, `writeProjectSystemPrompt`). Prompt composition in `src/agent/prompt.ts:buildSystemMessage` (`append=true` concatenates, `append=false` replaces). `hybridRecall`, `searchBM25`, `searchVector` all accept a `project` filter ŌĆö projects never leak into each other. Project name is immutable (PK + dir); description/visibility/languages/prompt can change. CLI `--project` auto-creates DB row + directory. See [ADR 0008](./docs/adr/0008-projects.md), [docs/dev/concepts/projects-as-scope.md](./docs/dev/concepts/projects-as-scope.md).

## Subsystems

Each stub is: **purpose ŌåÆ entry points ŌåÆ load-bearing invariant ŌåÆ docs**. Full walkthroughs live in the linked ADR + `docs/dev/` page.

### Agents

Named personalities with their own system prompt, tool whitelist, memory knobs. Orthogonal to Projects ŌĆö `agents` table + `$BUNNY_HOME/agents/<name>/config.toml` + `project_agents(project, agent)` join controls availability. `runAgent` merges memory knobs with precedence **agent ŌåÆ project ŌåÆ global**; `context_scope = "own"` passes the agent name as `ownAuthor` so recall filters to user + own rows. `POST /api/chat` takes optional `agent` field and **falls back to `cfg.agent.defaultAgent` when neither the body nor a leading `@mention` names one** ŌĆö every new turn resolves to a named agent. `bunny` (configurable) is seeded at boot by `src/memory/agents_seed.ts:ensureDefaultAgent` and auto-linked to every project (existing at boot, new via `POST /api/projects`). `src/agent/mention.ts:parseMention` only strips a leading `@name` when `getAgent(db, name)` returns a row (else the scanner in `src/notifications/mentions.ts` owns it ŌĆö see Notifications). Regenerate inherits the responding agent from the target row, falling back to the default for legacy NULL chains. Composer exposes a per-session agent picker persisted at `localStorage["bunny.activeAgent.<sessionId>"]`; sidebar "New chat withŌĆ”" opens a modal that mints a session pre-bound to the picked agent. Subagents: `is_subagent = true` + `allowed_subagents` list; `call_agent` spawns nested `runAgent` with silent renderer; capped at `MAX_AGENT_CALL_DEPTH = 2`. Entry points: `src/memory/agents.ts`, `src/memory/agents_seed.ts`, `src/memory/agent_assets.ts`, `src/tools/call_agent.ts`, `web/src/lib/messageLabel.ts`, `web/src/lib/activeAgent.ts`, `web/src/contexts/DefaultAgentContext.tsx`. See [ADR 0009](./docs/adr/0009-agents.md), [ADR 0031](./docs/adr/0031-every-chat-is-agent-bound.md), [docs/dev/entities/agents.md](./docs/dev/entities/agents.md).

### Skills

Reusable instruction packages following the [agentskills.io](https://agentskills.io) open standard. `$BUNNY_HOME/skills/<name>/SKILL.md` (YAML frontmatter ŌĆö **YAML here, TOML elsewhere**, per the standard). Progressive disclosure: catalog in system prompt ŌåÆ `activate_skill` tool loads full body ŌåÆ bundled scripts/references via `read_file`. Install from GitHub URLs / skills.sh identifiers via `POST /api/skills/install`. Entry points: `src/memory/skills.ts`, `src/memory/skill_assets.ts` (mtime-cached), `src/memory/skill_install.ts`, `src/tools/activate_skill.ts` (closure-bound, in `DYNAMIC_TOOL_NAMES`). See [ADR 0013](./docs/adr/0013-agent-skills.md), [docs/dev/entities/skills.md](./docs/dev/entities/skills.md).

### Chat (incl. Quick Chats, fork, edit, regenerate)

Live SSE via `fetch` body-reader (not `EventSource`). Admins get a "Mine / All" scope toggle (absorbs the former Messages tab). **Quick Chats** = per-user `is_quick_chat` flag on `session_visibility`, auto-hidden after 15 min by `session.hide_inactive_quick_chats`. **Fork to Quick Chat** copies non-trimmed history. Per-bubble edit / save+regenerate / fork. Regenerate-as-alt-version chains via `messages.regen_of_message_id`, navigated via `< n/m >`. **Chronological items timeline:** `web/src/hooks/useSSEChat.ts` collects every reasoning / content / tool / question segment into a `Turn.items[]` array in arrival order so multi-question turns don't clump every card at the top of the bubble; the live elapsed timer subtracts time spent paused on `ask_user` answers (`pausedAtMs` / `pausedTotalMs`). **Sticky-bottom autoscroll** ŌĆö `ChatTab` and `CodeChatView` only follow the stream while the viewport is already near the bottom; scrolling up to read or answer an off-screen question card unsticks, sending a new prompt re-arms stickiness. See [ADR 0023](./docs/adr/0023-chat-quick-chats-fork-edit-regen.md), [docs/dev/entities/chat.md](./docs/dev/entities/chat.md).

### Interactive user questions (`ask_user`)

Closure-bound tool (`src/tools/ask_user.ts`) ŌĆö emits `ask_user_question` SSE event, registers a pending promise in `src/agent/ask_user_registry.ts` (keyed `sessionId::questionId`), awaits it. Default timeout 15 min. Answers post to `POST /api/sessions/:sessionId/questions/:questionId/answer` (in `chat_routes.ts`), which resolves the in-memory waiter; 404 when stale. **Gated by `askUserEnabled`** (see table above). Web UI: `web/src/components/UserQuestionCard.tsx`. See [ADR 0026](./docs/adr/0026-ask-user-question-tool.md).

### Boards

Per-project Trello-style kanban. One board per project ŌĆö `project` is the scope key, no separate `boards` table. Three append-only tables: `board_swimlanes`, `board_cards`, `board_card_runs`. Sparse positions (steps of 100). `createProject` seeds Todo/Doing/Done. **Card assignee is mutually exclusive** ŌĆö `assignee_user_id` XOR `assignee_agent`. Agent-assigned cards execute via `runCard` (`src/board/run_card.ts`) ŌĆö detached `runAgent`, events mirrored into an in-memory fanout keyed by `runId`, final answer written via `markRunDone`; fanout replays to late subscribers, drops after 60 s. **Auto-run reservation is atomic via `clearAutoRun`** ŌĆö fires exactly once per reservation under concurrent ticks; handler `board.auto_run_scan` seeded `*/5 * * * *`. Agent tools (`src/tools/board.ts:makeBoardTools`, `BOARD_TOOL_NAMES`): six closure-bound tools with project + db + userId baked in (cross-project isolation). Permissions: view = `canSeeProject`; swimlane CRUD = admin/owner; card-create = any viewer; patch/move/archive/run = `canEditCard`. See [ADR 0010](./docs/adr/0010-project-boards.md), [docs/dev/entities/boards.md](./docs/dev/entities/boards.md).

### Workspaces (per-project files)

`<projectDir>/workspace/` seeded with `input/`, `output/`, `code/` by `ensureProjectDir` (idempotent backfill). Every path flows through `safeWorkspacePath` (rejects absolute paths, `..`-traversal, symlink escapes). **`input/`, `output/`, `code/` are protected roots ŌĆö delete/move on the roots refuses; contents are freely editable.** Agent tools (`src/tools/workspace.ts`, `WORKSPACE_TOOL_NAMES`): `list_workspace_files`, `read_workspace_file`, `write_workspace_file`. Reads capped (64 KB utf8 / 5 MB base64) with `truncated` + `returnedBytes` / `totalBytes`. Routes in `src/server/workspace_routes.ts`; read = `canSeeProject`, mutate = `canEditProject`. See [ADR 0012](./docs/adr/0012-project-workspaces.md), [docs/dev/entities/files.md](./docs/dev/entities/files.md).

### Web Tools

Closure-bound (`src/tools/web.ts`, `WEB_TOOL_NAMES`): `web_fetch` (markdown via `node-html-markdown`, 100 KB cap), `web_search` (**SERP API via serper.dev when `[web] serp_api_key` set / `SERP_API_KEY` env; falls back to DuckDuckGo scrape with 5-attempt exponential backoff, then Bing**), `web_download` (100 MB cap, writes via `writeWorkspaceFile`). `WebConfig` in `src/config.ts`. `parseDuckDuckGoResults` is exported for testability. ### Whiteboards

Per-project Excalidraw canvases. `whiteboards` table (elements JSON + thumbnail). Entry: `src/memory/whiteboards.ts`, `src/server/whiteboard_routes.ts`. Two LLM modes: **edit** (`POST /api/whiteboards/:id/edit` ŌĆö `runAgent` with `systemPromptOverride`, hidden session via `session_visibility`) and **question** (`POST /api/whiteboards/:id/ask` ŌĆö seeds a chat session with PNG attachment, returns `{ sessionId }`). Auto-saves debounced 2 s. Frontend dep: `@excalidraw/excalidraw`. See [ADR 0015](./docs/adr/0015-whiteboards.md), [docs/dev/entities/whiteboards.md](./docs/dev/entities/whiteboards.md).

### Documents

Per-project rich-text markdown. `documents` table (`id`, `project`, `name`, `content_md`, `thumbnail`, timestamps; `UNIQUE(project, name)`). Tiptap + `tiptap-markdown`. Ribbon (`DocumentRibbon.tsx`) + WYSIWYG/code toggle. Edit / question modes mirror whiteboards. Images uploaded via `POST /api/documents/:id/images` ŌåÆ `<projectDir>/workspace/documents/<docId>/images/<uuid>.<ext>`. Whiteboard embeds (live + static) via custom `whiteboardEmbed` Tiptap node. Exports: `.docx` (via `docx` npm), HTML zip (via `jszip`), PDF (`window.print()`). Entry: `src/memory/documents.ts`, `src/server/document_routes.ts`. ### Contacts

Per-project. `contacts` (JSON `emails`/`phones`/`tags`/`socials`, data-URL `avatar`) + `contact_groups` (`UNIQUE(project, name)`, optional `color`) + `contact_group_members` + M:N `contact_businesses(contact_id, business_id, role?, is_primary)`. vCard import (client-side parser `web/src/lib/vcard.ts`, no deps; recognises `URL`, `X-SOCIALPROFILE`, `IMPP`, legacy `X-TWITTER`/`X-LINKEDIN`/`X-GITHUB`) + export (server-side 3.0). Contact Picker API button on Android Chrome (feature-detected). Edit / question modes mirror documents. Each contact carries an LLM-curated **soul** (`soul`, `soul_status Ōłł {idle|refreshing|error}`, `soul_refreshed_at`, `soul_next_refresh_at`, `soul_sources`) refreshed periodically via `web_search`+`web_fetch` against the contact's socials + website. State machine + race-safe claim mirror `user_project_memory`. Cap `ENTITY_SOUL_CHAR_LIMIT = 4000` (separate from `MEMORY_FIELD_CHAR_LIMIT`). `contact_translations` source fields are now `notes` + `soul` (sidecar gained a nullable `soul TEXT`); soul stale-marking gated by `cfg.contacts.translateSoul`. The on-demand `POST .../:id/soul/refresh` SSE route shares one `refreshOneContactSoul` helper with the periodic handler. Entry: `src/memory/contacts.ts`, `src/contacts/{soul_refresh_handler,soul_sweep_stuck_handler}.ts`, `src/server/contact_routes.ts`. Queue topic: `contact`. See [ADR 0019](./docs/adr/0019-contacts.md), [ADR 0036](./docs/adr/0036-social-handles-and-businesses.md), [docs/dev/entities/contacts.md](./docs/dev/entities/contacts.md).

### Businesses

Sibling entity to Contacts (ADR 0036). Per-project organisation table (`name`, `domain`, `description`, `notes`, `website`, JSON `emails`/`phones`/`socials`/`tags`, data-URL `logo`) plus the same eight `soul_*` columns and a `source Ōłł {manual, auto_from_contacts}` provenance field. M:N to contacts via `contact_businesses` (helpers in `contacts.ts`). Race-safety for auto-build comes from two partial UNIQUE indexes ŌĆö `(project, lower(name)) WHERE deleted_at IS NULL` + `(project, domain) WHERE domain IS NOT NULL AND deleted_at IS NULL` ŌĆö paired with `INSERT ŌĆ” ON CONFLICT DO NOTHING RETURNING id` in `upsertBusinessByName`. Auto-build is **opt-in per project** via `projects.auto_build_businesses` (admin toggle); the `business.auto_build` handler walks alive contacts, extracts `(name, domain)` candidates from `company` + email/website domains, dedupes, inserts unseen rows + links them, and optionally enriches up to `cfg.businesses.autoBuildBatchSize` new rows per tick via a `web_search`-driven runAgent call. Soul refresh mirrors contacts (`business.soul_refresh`, `business.soul_sweep_stuck`). `business_translations` source fields are `description` + `notes` + `soul`. Two new agent tools `lookup_contact` + `lookup_business` (closure-bound, project-scoped; in `DYNAMIC_TOOL_NAMES`) let the agent pull entity profiles + soul on demand ŌĆö there is **no automatic soul injection in the system prompt**. Entry: `src/memory/businesses.ts`, `src/businesses/{soul_refresh_handler,soul_sweep_stuck_handler,auto_build_handler}.ts`, `src/server/business_routes.ts`, `src/tools/lookup_entity.ts`. Queue topic: `business`. See [ADR 0036](./docs/adr/0036-social-handles-and-businesses.md), [docs/dev/entities/businesses.md](./docs/dev/entities/businesses.md).

### Knowledge Base

Sub-tab shell. v1 sub-tab: **Definitions** ŌĆö per-project dictionary. `kb_definitions` (`UNIQUE(project, term)` with `term COLLATE NOCASE`): manual + LLM short/long/sources + `active_description Ōłł {manual, short, long}` + `is_project_dependent`. **State split:** `llm_cleared = 0` + NULL = never generated (auto-fill candidate); `llm_cleared = 1` + NULL = explicitly cleared (skip). Generation: `POST /.../generate` (SSE) flips `llm_status ŌåÆ 'generating'` (lost race ŌåÆ 409), hidden `kb-def-<uuid>` session, `runAgent` with `DEFINITION_SYSTEM_PROMPT` + `webCfg`; `try/catch/finally` guarantees the row never stays `generating`. Project-dependent mode blends term with `projects.description` before searching. **SVG illustrations:** parallel column set (`svg_*`), generated with `ILLUSTRATION_SYSTEM_PROMPT` and **no `webCfg`**; rendered via `<img src="data:image/svg+xml,${encodeURIComponent(svg)}">` so stray `<script>` is context-isolated (no sanitizer needed; no `;charset=` / `;utf8` parameter ŌĆö bare `;utf8,` is RFC 2397-malformed). **Stuck-row recovery:** `kb.sweep_stuck` (handler `src/kb/sweep_stuck_handler.ts`, cron `*/5 * * * *`) flips `llm_status` / `svg_status` rows still on `'generating'` after `updated_at < now - 30min` back to `'idle'` so `kb.auto_generate_scan` retries on the next tick (rescue path for process death mid-call). Queue topic: `kb`. Entry: `src/memory/kb_definitions.ts`, `src/server/kb_routes.ts`. See [ADR 0021](./docs/adr/0021-knowledge-base-definitions.md), [docs/dev/entities/knowledge-base.md](./docs/dev/entities/knowledge-base.md).

### Multi-language translation

Project `languages` + `default_language`; `users.preferred_language` overrides per view. Four sidecar tables (`kb_definition_translations`, `document_translations`, `contact_translations`, `board_card_translations`) with `status Ōłł {pending, translating, ready, error}`, `source_hash`, `translating_at`. Shared abstraction `src/memory/translatable.ts:registerKind({...})` per module. **Every source edit path must call `markAllStale` ŌĆö load-bearing; forget it and translations silently drift.** `create*` functions call `createTranslationSlots`. `backfillTranslationSlotsForProject` on `updateProject` language expansion + `backfillAllTranslationSlots` at boot (idempotent `ON CONFLICT DO NOTHING`). `aliveFilter` keeps trashed/archived entities out. **Staleness is two-coordinate:** `entity.source_version` (scheduler filter) + sidecar `source_hash` (editŌåÆrevert short-circuits to `ready` without an LLM call). Handlers: `translation.auto_translate_scan` (`*/5 * * * *`) + `translation.sweep_stuck` (`0 3 * * *`, reclaims rows stuck > `cfg.translation.stuckThresholdMs`). **KB short/long are translated, not regenerated** ŌĆö `llm_sources` stays on the entity row. **Orphaned languages are soft-kept** (removing a language leaves `isOrphaned=true` sidecars; re-adding resurfaces them). UI: `web/src/components/TranslationsPanel.tsx`; initial tab via `web/src/lib/resolveActiveLang.ts`. See [ADR 0022](./docs/adr/0022-multi-language-translation.md), [docs/dev/concepts/translation-pipeline.md](./docs/dev/concepts/translation-pipeline.md).

### Scheduler

Generic periodic-task subsystem in `src/scheduler/`. `scheduled_tasks` = single source of truth (`kind = system|user`, `handler`, `cron_expr`, `payload`, `enabled`, `owner_user_id`, `next_run_at`). `HandlerRegistry` ŌĆö domain modules self-register; the scheduler knows nothing about boards/agents/news/translation. Ticker runs once per minute (`src/scheduler/ticker.ts`), atomically claims due rows via `claimDueTasks` (bumps `next_run_at` by one minute in the same transaction), invokes, stores `setTaskResult` with real `computeNextRun(cron, now)`. **Malformed cron parks the row one hour out rather than crashing the tick.** System-tasks visible to all / admin-managed; user-tasks owned by creator (admins see all). HTTP: `/api/tasks*`. See [ADR 0011](./docs/adr/0011-scheduled-tasks.md), [docs/dev/concepts/scheduler.md](./docs/dev/concepts/scheduler.md).

### Notifications

Per-user (cross-project). v1 trigger: `@username` mentions in chat prompts. `notifications` with `ON DELETE CASCADE` on recipient; `actor_username` / `actor_display_name` denormalised (survives actor deletion); pruned to newest 200 per user per insert. Scanner `src/notifications/mentions.ts:parseUserMentions` walks the whole prompt (excludes emails, URLs, path-like `folder/@user`, code blocks/inline spans); dispatcher skips unknown + self; recipients who can't see the project get aggregated into a single `mention_blocked` counter-row. **Gated by `mentionsEnabled`.** **Leading-@ collision:** `handleChat` only strips a leading `@name` when `getAgent(db, name)` returns a row ŌĆö username-only leading tokens flow through to the scanner. Fanout (`src/notifications/fanout.ts`): in-memory map, 25 s keepalive, **no replay buffer** (new subscribers call `GET /api/notifications` for history). Logout ŌåÆ `closeAllFor(userId)`. Reading another user's notification returns **404, not 403** ŌĆö row existence stays private. SSE events: `notification_created`, `notification_read`. UI: `NotificationBell` (in `.nav__user-row` ŌĆö badge visible when rail collapsed) + `ToastStack` + `web/src/tabs/NotificationsTab.tsx` + `web/src/hooks/useNotifications.ts`. Toasts/OS pings suppressed when user is viewing target session. `osToast` feature-detects `window.__TAURI__` ŌåÆ `@tauri-apps/plugin-notification` vs `window.Notification`. Queue topic: `notification`. See [ADR 0027](./docs/adr/0027-user-notifications.md), [docs/dev/concepts/notifications-and-fanout.md](./docs/dev/concepts/notifications-and-fanout.md).

### Telegram

Per-project bot integration. Inbound DMs ŌåÆ `runAgent` like `/api/chat`; outbound hooks mirror `@mention` notifications, `card_run_finished`, Web News digests. Tables: `project_telegram_config` (`bot_token UNIQUE`, `transport Ōłł {poll, webhook}`, `webhook_secret`, `last_update_id`, `poll_lease_until`), `user_telegram_links` (PK `(user_id, project)` + UNIQUE `(project, chat_id)`), `telegram_pending_links` (15-min TTL), `telegram_seen_updates`, `web_news_topic_subscriptions`. **Linking is per-project because the bot is.** Inbound (`src/telegram/handle_update.ts`): dedup ŌåÆ **advance `last_update_id` BEFORE processing** (poison-message safety) ŌåÆ slash-commands ŌåÆ chat_id ŌåÆ user_id ŌåÆ per-chat mutex ŌåÆ `runAgent({ askUserEnabled: false, mentionsEnabled: true })` with `collectingRenderer` ŌåÆ format + chunk + send. Transport: poll default (handler `telegram.poll`, cron `* * * * *`, 50 s lease); webhook opt-in when `BUNNY_PUBLIC_BASE_URL` set ŌĆö `applyTransport` toggles so `getUpdates` and webhook never collide (Bot API 409 otherwise); `reapplyAllTransports` runs at boot. Outbound: **HTML subset** via `src/telegram/format.ts` (MarkdownV2's escape rules are a footgun), 4000-char chunking with `(n/m)` prefix, `sendDocument` fallback above 16 KB; rate limiter 30/s global + 1/s per chat. Hook points: `src/notifications/mentions.ts`, `src/board/run_card.ts`, `src/web_news/run_topic.ts`. **Webhook endpoint `POST /api/telegram/webhook/:project` is mounted BEFORE the auth middleware** in `routes.ts`; constant-time compare via `crypto.timingSafeEqual`; **always returns 200** (Telegram would otherwise retry deliberate rejects); dispatch detached. Authenticated surface = admin or project creator only (bot token is an impersonation capability; read returns `tokenTail` only; webhook secret never returned). **Tokens are never logged ŌĆö only `tokenTail` (last 4 chars).** Queue topic: `telegram`. See [ADR 0028](./docs/adr/0028-telegram-integration.md), [docs/dev/concepts/telegram-integration.md](./docs/dev/concepts/telegram-integration.md).

### Workflows

Per-project TOML-defined DAGs under `<projectDir>/workflows/<slug>.toml`. Three append-only tables: `workflows` (index + `toml_sha256` drift hash + `bash_approvals` per-node sha256 map + `layout_json` for xyflow positions), `workflow_runs` (frozen `toml_snapshot`, umbrella `session_id`), `workflow_run_nodes` (per-iteration; `iteration = 0` for non-loop). Four node kinds: `prompt` (runAgent), `bash` (gated shell), `loop` (iterate until `<<<until>>>` literal substring), `interactive` (stand-alone `ask_user_question` via existing registry). Engine `src/workflows/run_workflow.ts` mirrors `run_card.ts` ŌĆö detached runner + in-memory fanout with 60 s post-close TTL. **v1 is strictly serial** (topological walk). Loop stop-detection = magic-string sentinel, no new tool. Interactive gates reuse `ask_user_registry.ts` verbatim ŌĆö zero new SSE surface beyond the four workflow-lifecycle events (`workflow_run_started`, `workflow_node_started`, `workflow_node_finished`, `workflow_run_finished`). **Bash nodes are gated by four hard walls:** `[workflows] bash_enabled = false` default + first-run `ask_user` approval per `(workflow, nodeId)` recording `sha256(command)` + cwd = `safeWorkspacePath(project, ".")` + 120 s default / 600 s hard-max timeout + 256 KiB output cap + explicit env whitelist (never inherit `LLM_API_KEY`/admin passwords). ### Soft-delete and trash bin

Six trashable entities: `documents`, `whiteboards`, `contacts`, `kb_definitions`, `code_projects`, `workflows` (board cards use their own `archived_at`). `src/memory/trash.ts` exposes `registerTrashable({...})` + `softDelete` / `restore` / `hardDelete` / `listTrash`; each entity module self-registers on import (mirrors `registerKind`). Adding a seventh = one call + two columns. **UNIQUE(project, name|term) collisions avoided by renaming to `__trash:<id>:<original>` inside the soft-delete transaction.** Restore strips the prefix; clash with a live row returns `name_conflict` (HTTP 409). **Translation sidecars dropped on soft-delete and reseeded on restore** via `reseedTranslations` ŌåÆ `createTranslationSlots` (keeps `translatable.ts` ignorant of trash; avoids scheduler chasing ghosts). **Every list/get query must carry `AND deleted_at IS NULL`.** Audit grep: `FROM (documents|whiteboards|contacts|kb_definitions|code_projects|workflows)\b` ŌĆö zero hits without the predicate. Mangled `__trash:` names leaking into UI is the canary. HTTP (admin-only): `/api/trash`, `POST .../restore`, `DELETE .../`. Queue topic: `trash`. See [ADR 0025](./docs/adr/0025-soft-delete-and-trash.md), [docs/dev/concepts/soft-delete-and-trash.md](./docs/dev/concepts/soft-delete-and-trash.md).

## Conventions

- **TOML over YAML** for config; `.env` only for secrets. (Exception: `SKILL.md` uses YAML frontmatter because the agentskills.io standard does.)
- **English only, always.** Every artefact written to the repo or to GitHub must be in English ŌĆö no Dutch, no mixed language. Applies to commit messages, PR titles/descriptions/comments, all Markdown (`README.md`, `CLAUDE.md`, `docs/**`, ADRs), code identifiers + comments + log strings + error messages, test names, TOML/JSON `description` fields, seeded sample data. Chat replies to the user may follow the user's language ŌĆö but **nothing that lands in `git log`, the file tree, or GitHub** may.
- **Schema is append-only.** When changing `src/memory/schema.sql`, add new columns rather than altering existing ones ŌĆö state is long-lived in `$BUNNY_HOME`.
- **Every HTTP mutation logs through the queue.** Add `void ctx.queue.log({ topic, kind, userId, data })` after each successful write. `topic` = domain noun (`project`, `board`, `auth`, `agent`, `task`, `workspace`, `apikey`, `user`, `session`, `kb`, `web_news`, `translation`, `notification`, `telegram`, `code`, `trash`, `prompts`); `kind` = verb or dotted verb (`create`, `update`, `delete`, `card.move`, `login.failed`, `clone.error`). Always include `userId` when authenticated. **Never log secrets** (passwords, API-key values, bot tokens, webhook secrets ŌĆö only `tokenTail`). Queue is fire-and-forget (`void`) ŌĆö logging must never block the response. See [docs/dev/concepts/queue-and-logging.md](./docs/dev/concepts/queue-and-logging.md).
- **Provider-specific streaming quirks belong in `src/llm/profiles.ts`**, not `adapter.ts` or `stream.ts`.
- **`Renderer` optional methods are opt-in surfaces.** `onAskUserQuestion?`, `onQueueWait?`, `onQueueRelease?` may stay undefined on transports that don't surface them (silent / collecting renderers). The agent loop checks for presence with `?.`; never required to be implemented.
- Tests live under `tests/` mirroring `src/` layout. DB tests use `mkdtempSync` + `openDb(path)` for isolation.
- Design decisions are captured in `docs/adr/` (numbered). **Add a new ADR when making a non-trivial architectural choice.**
- **Visual language is canonised in [`docs/styleguide.md`](./docs/styleguide.md)** ŌĆö tokens, spacing scale, icon sanctioning via `lucide-react` + `web/src/lib/icons.ts`, rabbit placements, shared primitives. Update the styleguide in the same PR that shifts it and add a dated change-log entry.
- **Dev handbook:** narrative orientation, per-entity walkthroughs, per-UI-surface reference, and extension recipes live in [`docs/dev/`](./docs/dev/) (`getting-started/`, `concepts/`, `ui/`, `entities/`, `how-to/`, `reference/`). Ship docs with code: if a PR changes a subsystem, the matching `docs/dev/` page updates in the same PR.
- **Before every commit:**
  - Run `bun test` ŌĆö any broken or newly-uncovered module gets a test in `tests/` mirroring the source path.
  - Update `README.md` if the user-facing workflow changed (new commands, flags, runtime requirements).
  - Update `docs/README.md` and add/amend an ADR in `docs/adr/` for non-trivial architectural changes.
  - Update the matching `docs/dev/` page.
  - Update this `CLAUDE.md` when **conventions, build steps, or the high-level architecture** shift.
  - Do not commit if tests regress or if a user-visible change has no accompanying doc update ŌĆö fix first, commit after.
